【系列连载二】引擎点火：驯服 Hugo 引擎的“底层洁癖”与跨目录挂载

哈喽，各位极客小伙伴们！ 在上篇连载中，咱们成功用 Filebrowser 搭建了一个 【零摩擦的本地网盘】。现在，你的 Markdown 笔记已经可以随手拖拽进 Ubuntu 虚拟机里了。 但网盘里的 .md 文件终究只是“毛坯房”，咱们需要一台强悍的渲染引擎，把它们瞬间编译成带有暗黑模式、目录导航的精美极客网页。俺选用的引擎是目前地表最快的静态生成器—— 【Hugo】。 不过，Hugo 出了名的脾气古怪，有着极度的 【底层洁癖】。今天，俺就带大伙儿扒一扒它的底裤，看看如何绕过它设下的重重陷阱！ ★ 避雷指南：千万别用 Snap 装 Hugo 在 Ubuntu 上装软件，大伙儿的第一反应肯定是敲 sudo apt install 或者用系统自带的 snap 商店。 听俺一句劝：【绝对不要用 Snap 安装 Hugo！】 Snap 本质上也是一个极其严格的 【沙盒环境】。如果你用 Snap 装了 Hugo，当 Hugo 试图去读取咱们挂载在桌面（~/Desktop/）的网盘文件时，会直接被操作系统的安全机制“无情绞杀”，报出一堆莫名其妙的 Permission Denied（权限拒绝）。 【架构师的解法】：直接去 Hugo 的 GitHub Releases 页面，下载最新的 Linux tar.gz 压缩包，解压后把二进制文件直接扔进 /usr/local/bin/。绕过所有中间商，把引擎的控制权 100% 握在自己手里！ ★ 跨越物理隔离：神奇的“虚拟挂载” Hugo 的“洁癖”体现在它的目录结构上：它要求所有的文章必须乖乖放在它自己生成的 content/posts/ 文件夹下。 但这违背了咱们的自动化初衷！咱们的文章明明放在上一篇建好的 SyncDrive 网盘里。如果每次都要复制文件，那还搞啥自动化？如果用 Linux 的软链接（ln -s），又容易遇到各类缓存同步的诡异 Bug。 这里，俺要祭出 Hugo 的一个高级黑魔法：【模块虚拟挂载 (Module Mounts)】。 ...
